Overview

Prior Work

This site is a full implementation of the Wdd130 activity scheduler site

Purpose

The purpose of this site is to coordinate schedules with friends and acquaintances, to allow any member of a group to propose an activity that every member can attend, and to carry notifications of said proposals to other members of the group.

The website is not a general communications application. Its utility is limited to when activities are scheduled, not what those activities are.

Audience

The audience consists of everyone 12 and older. The idea is to reach out to groups of casually connected individuals, rather than formal clubs where people are free to join or leave. Most people may interact with this on a mobile device. Some of the group meetings may be between different kinds of people. For groups of around 4-5.

Content

The site requires five things:

  1. Login
  2. Other User Authentication
  3. Rules for availability
  4. The activity scheduler module
  5. Event center

These are defined as follows

Login

Not to be implemented at this point in time. However, each user needs to have their own account. Each account will need to have the following information:

  1. Their schedule
  2. List of users who they have authorized to see their schedule and send them notifications
  3. List of users who have authorized them
  4. The activity scheduler module
  5. Events received

There needs to be an account page, as well as a login page. The account page at least needs a logout and a delete account. More complicated account proceedures include changing passwords and protection against destructive actions (like accidentally deleting account). These can be implemented if time permits.

Other User Authentication

This represents a place where users can authorize other users to contact them and view their schedule. At the top, we have an image for the user and their username. Then we text input for the name of the user to add and a button to add them. Below that we have of list of authorized individuals each one as follows:

  • The username
  • An icon to represent whether they've reciprocated.
  • A checkbox to authorize or de-authorize that individual
  • A deletion button.

Rules for availability

A rule is composed of the following:

  • A start date and an end date
  • Whether it is on or off on each day of the week
  • Whether it represents available time or unavailable time.
  • A start time hr:mn:am/pm and an end time

The rule page will be composed of a button to add rules, and a list of rules such that

  • Drop down menus for the year, month, and day of the start date and also drop down menus for the year month, and day of the end date
  • Drop down menus for start hour, 30 minute interval, with am/pm being kept track by using 24-hour time, and also drop down menus for the end time. (Updated)
  • A checkbox to indicate whether the rule is active
  • A delete icon to remove the rule (ideally should pop up confirmation window)

Event Center

This page will have a list of activities, where each activity will have the following.

  • Date
  • Start time and end time
  • Name or description.
  • Names of other participants.
  • Delete button

Actitity Scheduler Module

This represents a place where people actually see availability and propose activities. Use requires adding all group members, setting the date, and scrolling down to the available time.

  • Scroll listing group members checkbox if in group
  • Scroll for times at 30 minute intervals, colored differently based on availability
  • Clicking or tapping on time brings up activity schedule form

Responsiveness

Login page

The natural starting place is the login page. Logging in should take the user to the event center page. When a user creates an account, send them to the rules page. If logged in on page, user should remain logged in when clicking on in-site links. When logged in, each page should start by bringing in the relavant data.

Account page

The account page has two buttons. The logout button should logout the user and return the user to the login page. The remove account button should remove the user's account from the database, remove all references to that user in "other user authentication" and the associated section of the activity scheduler page. It should not remove the references in the event center.

Other User Authorization Page

Upon loading, all list entries should be checked against the database. All entries that reflect non-existent users should be removed. For all valid entries, "authorized by them" reflects whether that user has this user listed and authorized in the appropriate section.

When add user is hit, the name should be checked against the database, and if it is present, a new entry should be added, with "authorized by you" reflecting user input and "authorized by them" reflecting the value retrieved from the database. The database should be updated to reflect the new authorization.

Checking or unchecking "authorized by you" on an established list entry should be reflected in the database.

Hitting the remove button on any user should remove that user from the list and should be reflected in the database.

Rules for Availability Page

Hitting the "new" button should create a new entry on the list and the database. Changes on each rule should be reflected in the database. Hitting the delete button on a rule should remove it from the list and the database.

Activity Scheduler Page

List on top comes from other authorized users in the user account that are authorized both ways. All deselected by default: changes here alter highlights below.

User selecting date icon filters the display below to that date. User selecting time icon causes display to start at that time.

Each time in the timescroll portion to be highlighted if it lies within the user's rules and all of the selected users' rules. Selecting the plus icon within one of these highlighted times brings up the add event pop-up.

If the "create event" button is hit in the add event pop-up, an event object will be created with the given specifications and added to the user's event list and the selected other users' event lists.

Event Center

During loading, events should be tested to see if they are over. If they are over, they should be removed from the list and the user account.

When a page is loading, if a user has unread events, the event center link in the nav should highlight. If a user has events in the event center, but all are read, the event center link should highlight a different way.

Clicking the delete icon on an event removes that event from the list and the user account.
